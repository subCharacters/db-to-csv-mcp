## 코멘트 스타일
- 원칙: **무엇**은 코드, **왜/의도/제약**은 주석
- 클래스 주석은 항상 아래와 같은 포맷으로 작성한다.
### 클래스 주석 템플릿
```java
/**
 * 클래스의 책임(왜 존재하는가)을 1~2문장으로 기술.
 *
 * <p>주요 제약/성능 고려/스레드-세이프 여부 등 핵심 메모.
 *
 * @author 이름
 * @since 1.0
 * @version 1.0
 */
public class ExampleService {
}
```

- 메서드의 주석은 항상 아래와 같은 포맷으로 작성한다.
### 메서드 주석 템플릿
```java
/**
 * 무엇을 수행하고, 왜 이 접근을 택했는지 1~2문장으로 설명.
 *
 * <p>입출력 예시(필요 시):
 * input: "ABC", 3  -> output: "ABCABCABC"
 *
 * @param s 입력 문자열(널 불가)
 * @param times 반복 횟수(0 이상)
 * @return 반복된 문자열
 * @throws IllegalArgumentException times < 0 인 경우
 */
public String repeat(String s, int times) {
    // …
}
```

- 미시적 로직 주석은 지양. 다만 수식/알고리즘/조합 로직은 입출력 예시를 블록으로 남김.

## 테스트 원칙
- 핵심 비즈니스 로직은 커버리지 80%이상을 목표로한다.
- 단위 테스트는 필요한 부분만 가져와 빠르게 테스트 가능하게 한다.
- 다른 단위 테스트에 영향 없이 독립적으로 수행 하게 한다.
- 반복적으로 수행해도 같은 결과가 나오게 한다.
- assert를 사용하여 스스로 검증한다.
- 단위 테스트는 즉시 작성한다.
- 필수 케이스: 경계값/에러/동시성

## 커밋 컨벤션

### 규칙 요약
- 형식: `type(scope): subject` → 한 줄 비우고 → `body` → 한 줄 비우고 → `footer`
- 제목(subject) 최대 50자, 본문(body) 줄바꿈 72자 기준, 제목 뒤 공백 한 줄 필수.
- 제목은 **명령문**, **마침표/특수기호 금지**, **영문일 때 첫 글자 대문자**.
- 본문은 **무엇/왜** 중심(어떻게는 최소화), 항목이 많으면 `-`로 구분.
- **BREAKING CHANGE:**는 본문 또는 꼬릿말에 명시(주버전 연동). 릴리스 자동화/변경이력과 연계 권장.

### 타입 목록(팀 확장 포함)
- 표준: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `build`, `ci`, `chore`, `perf`, `revert`
- 팀 확장: `design`(UI/CSS), `comment`(주석만), `init`(초기화), `rename`(이름/경로 변경만), `remove`(삭제만)

### 템플릿
```text
<type>(<scope>): <Subject in imperative, <=50 chars>

<body: wrap at 72 cols>
- 무엇을 변경했는지
- 왜 변경했는지(배경/이유)
- 부작용/마이그레이션(있다면)

<footer>
BREAKING CHANGE: <호환성 깨짐 요약, 대안/마이그레이션>
Resolves: #123
Related to: #45, #67
```
### 예시
```text
feat(api): Add bulk export endpoint

- Introduce /v1/exports for CSV/JSON bulk download
- Reduce client round-trips; aligns with reporting requirements

Resolves: #231
```

```text
fix(db): Prevent NPE on null payload

- Validate request body before mapping
- Logs 400 with error code APP-4001
```

```text
refactor(core): Extract tokenizer as strategy
```

```text
chore(ci): Enable commit lint + changelog generation

- Enforce Conventional Commits at PR
- Auto-generate CHANGELOG.md on release
```


### 꼬릿말(footer) 가이드
- `Resolves: #<id>`: 이슈 해결 시  
- `Fixes: #<id>`: 해결 중(미완)  
- `Related to: #<id>`: 연관 이슈  
- `BREAKING CHANGE:`: 호환성 파괴 변경 필수 표기(주버전 증가/릴리스 노트 강조).

### 운영 팁(선택)
- **commitlint + husky**로 규칙 자동 검증
- **standard-version** 또는 **release-please**로 버전/CHANGELOG 자동화
- Conventional Commits ↔ SemVer/Keep a Changelog 연계
